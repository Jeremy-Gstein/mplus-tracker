MPTracker API

REST API for tracking Mythic+ keystones run by guild members. Queries Raider.IO for character and guild data, stores runs in SQLite, and exposes counts and history for external consumers (WeakAuras, Discord bots, dashboards).

Base URL: http://localhost:8080
Format: application/json
Auth: none
Storage: SQLite (WAL)
Errors & Common Concepts

All error responses use the same envelope. The scope parameter controls the time window for all query endpoints and is region-aware for weekly resets.

Error envelope — every non-2xx response returns:
{ "error": "Missing `realm` parameter", "code": 400 }
Scope values
ValueWindowNotes
today UTC midnight → now Calendar day in UTC
week Last weekly reset → now NA: Tue 15:00 UTC · EU: Wed 07:00 UTC. Pass region=eu to use EU reset.
alltime Epoch → now Default when scope is omitted
custom from → to Requires both from and to as RFC3339 strings
Deduplication

Every run is assigned a deterministic hash derived from region, realm, character name, dungeon, level, timestamp, and source run ID. INSERT … ON CONFLICT (hash) DO NOTHING means re-crawling a character never creates duplicate rows — only genuinely new runs are inserted.

Stale-crawl skip

When updating a guild, members whose Raider.IO last_crawled_at is within the last 3 months are skipped for individual key updates (they are still upserted for roster tracking). This dramatically reduces rate-limit pressure on large guilds.

Updates

Update endpoints pull fresh data from Raider.IO and persist it to the local database. All updates are idempotent — re-running them only inserts genuinely new runs.

POST /update/guild Sync guild roster + queue member key updates ▶

Fetches the guild profile from Raider.IO, upserts all members into the characters table, and queues individual key updates for members whose Raider.IO profile is stale (older than 3 months). Members crawled recently are upserted to keep the roster current but their key fetch is skipped.

Query parameters
ParamTypeRequiredDescription
region string required WoW region — us eu kr tw
realm string required Realm name, e.g. Stormrage
name string required Guild name, e.g. Seems Good
200 OK 400 Bad Request 500 Server Error
Response schema
request_idstring (uuid)Unique ID for this request
guild.regionstringEchoed from input
guild.realmstringEchoed from input
guild.namestringEchoed from input
members_addedintegerNew characters inserted this run
members_updatedintegerExisting characters refreshed
members_skipped_freshintegerMembers skipped — crawled by RIO recently
Example response
{ "request_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890", "guild": { "region": "us", "realm": "Stormrage", "name": "Seems Good" }, "members_added": 3, "members_updated": 412, "members_skipped_fresh": 103 }
Example request
curl -X POST 'http://localhost:8080/update/guild?region=us&realm=Stormrage&name=Seems%20Good'
POST /update/character Sync one character's recent runs from Raider.IO ▶

Fetches mythic_plus_recent_runs for the given character from Raider.IO and inserts any new runs into the database. Existing runs (identified by hash) are silently skipped — the count is reflected in runs_ignored.

Query parameters

character.realmstring
character.namestring

POST /update/all Sync all known characters concurrently ▶
⚠ High load: This triggers a Raider.IO fetch for every character in the database, bounded by the semaphore concurrency limit in config.toml. On large guilds this takes several minutes and burns significant rate-limit budget.

Spawns a Tokio task per character, bounded by the shared semaphore. Returns aggregate counts when all tasks complete. Individual failures are collected into errors and do not abort the batch.

200 OK 500 Server Error
Response schema
request_idstring (uuid)
total_charactersintegerTotal characters attempted
updated_okintegerSuccessfully fetched + synced
failedintegerCharacters that errored (network, RIO 404, etc.)
errorsstring[]Error messages for each failure
Example response
{ "request_id": "f1e2d3c4-b5a6-0987-fedc-ba9876543210", "total_characters": 518, "updated_ok": 515, "failed": 3, "errors": [ "Oldchar: RIO returned 404", "Xfer: request timeout after 10s", "Bankalt: RIO returned 404" ] }
Example request
curl -X POST http://localhost:8080/update/all
Queries

Read-only endpoints for fetching run counts and roster data. The scope parameter controls the time window. Weekly reset boundaries are region-aware (NA vs EU).

GET /character/:region/:realm/:name/keys Run count for a single character ▶

Returns the number of Mythic+ runs completed by the character within the requested time window. The character must already exist in the local database (run POST /update/character or POST /update/guild first).

Debug Endpoints

These endpoints expose raw DB contents for development, debugging, and verifying data integrity. They are not intended for production consumers — consider restricting them behind a reverse-proxy or firewall rule in prod.

⚠ Not for production consumers. These endpoints do full table scans and return potentially large payloads. Protect with a firewall or internal-only routing in production.
GET /debug/stats Aggregate database statistics and distributions ▶

Returns row counts, today/week run totals, dungeon distribution, key level histogram, and top-20 characters by run count. Useful as a health dashboard and sanity check after crawls.

200 OK
Response schema

total_runsintegerRows in runs table

runs_by_keylevel[]object[]{ key_level: integer, count: integer }
top_characters[]object[]{ name, realm, region, run_count } — top 20
Example request
curl http://localhost:8080/debug/stats

GET /debug/hash-check Verify whether a specific run hash is in the database ▶

Point lookup against the runs.hash unique index. Use this to confirm that a specific run you expected to be inserted actually made it in, or to verify that re-running an update would correctly skip a known run as a duplicate.

Query parameters
ParamTypeRequiredDescription
hashstringrequiredThe run hash to look up (from /debug/runs hash field)
200 OK
Response schema
hashstringEchoed from query param
existsbooleanTrue = run is in DB · False = would be a new insert
Example response — exists
{ "hash": "sha256:abc123...", "exists": true }
Example response — not found
{ "hash": "sha256:xyz789...", "exists": false }
Example request
curl 'http://localhost:8080/debug/hash-check?hash=sha256:abc123...'
